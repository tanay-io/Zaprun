Phase 5 responsibilities:

1. Persist execution state durably
   - ZapRun and StepState always reflect real progress
   - No progress exists only in memory

2. Persist execution intent via Outbox
   - Each step execution is scheduled through an outbox record
   - Outbox is the single source of truth for "what should run next"

3. Ensure single-worker execution using leasing
   - At any moment, only one worker can own an outbox job
   - Ownership is time-bound to allow recovery after crashes  jaise hmne os me 
   kra tha race conditions ke lie. 

4. Support at-least-once execution with safe retries
   - Steps may execute more than once
   - Runner controls retries with limits and backoff
   - Executors remain stateless

5. Prepare for replay by persisting step outputs
   - Successful step outputs are stored durably
   - Completed steps are never re-executed by default

6. Apply backpressure at scheduling time
   - If the system is overloaded, do not enqueue more work
   - Mark ZapRuns as queued/throttled instead of flooding the queue
